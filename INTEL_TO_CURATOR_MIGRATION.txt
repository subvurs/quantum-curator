1. WHAT WE ARE DOING
--------------------------------------------------------------------------------

Migrating the Quantum Intel synthesis pipeline (today living at
~/Library/Application Support/quantum_intel/) into the Quantum Curator
backend (~/Desktop/quantum-curator/) as a post-processing step that
runs after Curator's existing daily intake.

After migration:
  - Curator runs once per day as today (intake + curate + post to
    Quantum Crier + Qrater + email + per-article Bluesky)
  - As a NEW last step, Curator runs the (re-implemented) Intel
    synthesis pipeline over its own articles + the migrated history
  - Intel sends ITS OWN email (separate SMTP, separate body) with
    daily briefs + day-over-day summary
  - Curator's email gets a NEW subvurs_impact_score section
  - Curator posts a NEW daily Bluesky summary (less detailed than email,
    links to qrater.org)
  - Intel's standalone launchd cron is retired
  - Intel's briefs/ directory is preserved as a read-only historical
    archive

2. WHY
--------------------------------------------------------------------------------

The Intel pipeline has stalled chronically since at least April 2026.
Documented multi-hour hangs from review.log + run_log.json:

  - Apr 24 2026  review file not created (claude rc=0)
  - May 2  2026  review file not created (claude rc=1)
  - May 16-28    multiple rc=1 review failures
  - May 20 2026  single doc consumed 4h35m wall clock (catalog deadline
                 exception was being swallowed by chat_with_retry's
                 except RuntimeError, sleeping, then re-attempting)
  - May 21 2026  1h18m stall in collectors (DuckDuckGo CAPTCHA scrape)
  - Jun 7  2026  5h25m hang in SMTP (no socket timeout)
  - Jun 8  2026  synthesize 6h 8m, collect 57m, traced to PEP 475
                 EINTR-restart silently defeating signal.SIGALRM
                 contextmanagers in synthesizer/cataloger/collectors

Root causes consolidated into four classes:

  (a) PEP 475 EINTR auto-retry defeating SIGALRM-based deadlines.
      The alarm fires, the handler runs, but the very next syscall
      picks up where it left off (recv() inside resp.read(), or
      Popen.communicate() pipe reads) and the deadline does nothing.
  (b) llm_client falling back to Ollama Cloud when ANTHROPIC_API_KEY
      and NVIDIA_API_KEY are missing from the launchd environment.
      Ollama Cloud slow-drips bytes on a 600s urlopen timeout — the
      socket never goes idle, the per-socket timeout never fires.
  (c) Multi-provider fallback chain (Anthropic → NIM → Ollama local →
      Ollama Cloud) compounding any single provider's failure into
      cascading retries.
  (d) Duplicate scoring infrastructure: Intel has its own (recently
      added, Jun 2026) Stage 2.5 subvurs_impact scorer that fails
      closed when ANTHROPIC_API_KEY isn't set. Curator ALREADY has
      working Subvurs-impact scoring (loads .env via pydantic-settings,
      writes to curated_posts.subvurs_impact_score).

Migrating into Curator escapes all four:

  (a) Curator is async-native (httpx, not urllib). No PEP 475 issue.
  (b) Curator's pydantic-settings loads .env at startup — keys are
      reliably present.
  (c) Curator already uses the Anthropic SDK directly. No fallback
      chain to compound.
  (d) Move scoring to a single place that already works.

4. ARCHITECTURE — BEFORE vs AFTER
--------------------------------------------------------------------------------

BEFORE (today)

  Curator (cron via .github/workflows/daily-curator.yml)
    └── intake (arXiv quant-ph, ~17 RSS, NewsAPI if key)
    └── curate (LLM commentary + Phase A subvurs_notes string +
                Phase B subvurs_impact_score on curated_posts)
    └── publish (Quantum Crier site + Qrater site)
    └── email (insights_report — subvurs_notes only)
    └── Bluesky (per-article share)

  Intel (launchd cron, separate venv, separate .env)
    └── COLLECT (curator feed @ qrater.org, arXiv quant-ph + cond-mat,
                 DDG Lite 13 queries)
    └── CATALOG (LLM structured extraction, SHA-256 dedup)
    └── SUBVURS_SCORE (Stage 2.5, recently added, ~0.7% coverage so far)
    └── SYNTHESIZE (combinatorial-product brief generation)
    └── DELIVER (writes .md briefs to ~/Library/.../briefs/)
    └── SCAFFOLD (off by default since v1.6)
    └── EMAIL (SMTP brief delivery to Mark)
    └── REVIEW (separate claude CLI subprocess, frequently rc=1)

AFTER (target — REVISED Jun 9 2026 per Phase 5 plan)

  Curator (cron, same workflow file, ONE workflow)
    └── intake (arXiv quant-ph + cond-mat, ~18 RSS + Quantinuum + PRX
                Quantum, NewsAPI if key)  → raw_articles
    └── curate (unchanged — LLM commentary + Phase A + Phase B scoring)
                                          → curated_posts
    └── publish (Quantum Crier + Qrater — unchanged)
    └── email (insights_report + NEW subvurs_impact_score section)
    └── Bluesky (per-article share + NEW daily summary post linking
                 qrater.org — SHORT public rundown, distinct from
                 Intel email)
    └── INTEL SYNTH (new last step — Plan B sourcing)
        ├── TODAY'S SEEDS: pull from curated_posts where
        │   published_at ≥ now − 1d (Curator's own intake; the same
        │   articles that just went to Quantum Crier)
        ├── HISTORICAL CO-SOURCE: pull from quantum_intel_entries
        │   (1216 historical + any future entries that accrete via
        │   shim, see §5 Phase 5b option). These are NOT background —
        │   they are paired combinatorially with today's seeds.
        ├── combinatorial synthesis over (today × historical) pairs;
        │   anti-recurrence via recent brief citations (unchanged
        │   semantics — first_brief_at on quantum_intel_entries,
        │   plus parallel tracking on curated_posts.intel_first_brief_at)
        ├── daily summary (TL;DR + implications-vs-prior + attention
        │   recommendations, Goodhart-guarded; reads BOTH sources)
        ├── SEPARATE Intel email (own SMTP body — detailed internal
        │   review, distinct from Curator insights email AND distinct
        │   from Bluesky daily summary post)
        └── (no scaffold, no review — both dropped per Mark Jun 9 2026)

  Output channel separation (clarification Jun 9 2026):
    - Curator insights email = subvurs_notes + top-N subvurs_impact
                               table (on curated_posts only)
    - Intel email             = full brief content from synthesis
                                (curated_posts × quantum_intel_entries)
    - Bluesky daily summary   = short public rundown
                                (links → qrater.org)
    Three distinct outputs, three different shapes, three different
    audiences. None of them duplicate each other's content.

  Intel (~/Library/Application Support/quantum_intel/)
    └── launchd cron disabled
    └── inventory.json kept frozen as historical artifact (already
        backed up Jun 9 2026)
    └── briefs/ kept read-only
    └── COLLECT / CATALOG / SUBVURS_SCORE / SYNTHESIZE / DELIVER /
        EMAIL — all deprecated; module files left in place but unused
    └── pipeline_deadline.py retained (good engineering reference)

7. RISKS + MITIGATIONS
--------------------------------------------------------------------------------

  R1. Curator DB corruption during import.
      Mitigation: backup of inventory.json before any write; SQLite
      INSERT OR IGNORE idempotence; pre-import row counts captured
      for parity check (sources=18, raw_articles=448, curated_posts=28,
      etc.). Re-running the importer is a no-op. (Verified Jun 9.)

  R2. Curator publication path breakage.
      Mitigation: Phase 1 touched ONLY new tables and additive source
      entries. Existing tables unchanged. Existing publish path
      (publisher.py, site/, qrater_builder.py, email_report.py,
      bluesky.py) not touched in Phase 1.

  R3. Intel cron continues running and double-writes during parallel-run.
      Mitigation: Intel's cron writes to ITS OWN inventory.json — no
      shared writer. Curator-side synth reads quantum_intel_entries
      from Curator DB. The two pipelines are non-overlapping.

  R4. Anti-recurrence breakage in the new synthesizer.
      Mitigation: Phase 2 implementation preserves
      recent_brief_citations() semantics by reading from the new
      first_brief_at column on quantum_intel_entries. Golden-replay
      test against a small sample of Intel's existing briefs before
      cutover.

  R5. Goodhart on the daily summary LLM call.
      Mitigation: structured JSON output with explicit
      "no novel claims" constraint, fail-closed if response is not
      parseable JSON. Track wall-clock vs token count to detect
      slow-drift.

  R6. Schema drift between Intel inventory.json and the SQLite table.
      Mitigation: imported_from audit column records provenance;
      INSERT OR IGNORE means new fields in future Intel-format
      documents are silently dropped — log a warning if any.

  R7. (Phase 5) Output channel cross-contamination — Intel brief
      bodies leaking into Bluesky (300-char limit, gets truncated
      and looks broken) or into the Curator insights email
      (duplicates content).
      Mitigation: 5e smoke test in CI logs. Bluesky payload is
      built ONLY from daily_summary.render_bluesky(), which is
      explicitly short-form. Intel email payload is built ONLY
      from emailer.render_intel_email(). Curator insights email
      payload is built ONLY from email_report.build_insights
      _report() (reads curated_posts.subvurs_notes /
      subvurs_impact_score; never reads briefs). No shared
      string concatenation across these three.

  R8. (Phase 5) Goodhart on "non-empty for 3 days" cutover gate.
      If the synthesizer produces low-quality briefs to clear the
      gate, we cut Intel over to a worse pipeline.
      Mitigation: gate is "≥ 5 briefs/day with non-empty body
      AND at least one ≥ 0.3 subvurs_impact_score reference in
      either the seed or the paired historical entry" — not just
      brief count. Mark eyeballs the first 3 days' Intel emails
      before authorizing 4c unload.
